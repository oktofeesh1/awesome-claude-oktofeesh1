  ## Module System

  - Organize the application around feature modules; each module encapsulates
    a controller, service, and the providers specific to one domain
  - Use `@Global()` sparingly and only for truly application-wide providers
    such as a configuration service or logger
  - Avoid circular module dependencies; restructure shared logic into a
    dedicated shared module if two feature modules need each other

  ## Providers and Dependency Injection

  - Declare services, repositories, and helpers as `@Injectable()` providers
    registered in the module's `providers` array
  - Use constructor injection for all dependencies; avoid property injection
    except when extending a base class prevents constructor injection
  - Use the `@Inject(TOKEN)` decorator with a custom provider token for
    value providers, factory providers, and existing providers
  - Prefer `useClass` or `useFactory` for providers that need async
    initialization (e.g., database connections with `async useFactory`)

  ## Controllers and Request Handling

  - Keep controllers thin: parse and validate the request, delegate to a
    service, return the result
  - Apply method-level `@HttpCode()` and `@Header()` decorators when the
    default status code or headers do not match the operation semantics
  - Use `@Param()`, `@Query()`, `@Body()`, and `@Headers()` decorators with
    explicit types; never cast `req.params` manually

  ## Validation and Pipes

  - Apply `ValidationPipe` globally with `transform: true` and
    `whitelist: true` to strip unknown properties automatically
  - Define request body shapes with class-validator decorated DTOs; avoid
    plain interface types for request bodies that need runtime validation
  - Use `ParseIntPipe`, `ParseUUIDPipe`, and similar built-in pipes on
    route parameter decorators to validate path segments at the pipe layer

  ## Guards, Interceptors, and Exception Filters

  - Implement authentication with a `JwtAuthGuard` extending `AuthGuard`
    from `@nestjs/passport`; apply it globally and use `@Public()` to opt
    out specific routes
  - Use interceptors for cross-cutting concerns: logging, response
    transformation, and caching; keep business logic out of interceptors
  - Define an `AllExceptionsFilter` extending `BaseExceptionFilter` to
    translate unhandled errors into RFC-compliant problem responses

  ## OpenAPI Documentation

  - Decorate DTOs with `@ApiProperty()` from `@nestjs/swagger`; use
    `@ApiTags()` and `@ApiOperation()` on controllers for generated docs
  - Set `swaggerOptions: { persistAuthorization: true }` in the Swagger
    setup to preserve auth headers across page reloads in development

  ## Microservices

  - Use `@nestjs/microservices` transport layers (TCP, NATS, Redis, Kafka)
    for inter-service communication; choose the transport based on delivery
    guarantees and latency requirements
  - Use `@MessagePattern()` for request–response and `@EventPattern()` for
    fire-and-forget event handling in microservice controllers
  - Apply a `ClientProxy` injection via `ClientsModule.register()` for
    outbound microservice calls; always `connect()` and `close()` the proxy
    in lifecycle hooks

  ## Testing

  - Use `Test.createTestingModule()` to build isolated unit tests for
    services and controllers with mocked providers
  - Use `@nestjs/testing` `SuperTest` integration with `app.getHttpServer()`
    for end-to-end tests that exercise the full request pipeline

## Usage

Add this rule set to your project's `CLAUDE.md` to configure Claude as a NestJS expert. It covers the module system, constructor-injected providers, request pipeline (controllers, pipes, guards, interceptors, exception filters), OpenAPI documentation with `@nestjs/swagger`, microservice transport layers, and testing with `Test.createTestingModule()` — all grounded in the [official NestJS documentation](https://docs.nestjs.com/).

## Example: module, controller, and constructor DI

```typescript
// users.module.ts — feature module wiring a controller to its provider
@Module({
  controllers: [UsersController],
  providers: [UsersService],
})
export class UsersModule {}

// users.controller.ts — thin controller, constructor injection, pipe-validated param
@Controller('users')
export class UsersController {
  constructor(private readonly users: UsersService) {}

  @Get(':id')
  findOne(@Param('id', ParseIntPipe) id: number) {
    return this.users.findOne(id);
  }
}
```

## Request pipeline at a glance

Guards, pipes, interceptors, and exception filters each own one stage of the request lifecycle. Knowing which to reach for keeps logic in the right layer:

| Building block | Responsibility | Where it runs | Interface |
| --- | --- | --- | --- |
| Guard | Decide whether a request may proceed (authorization) | After middleware, before interceptors and pipes | `CanActivate` |
| Interceptor | Cross-cutting logic that wraps the handler (logging, transforms, caching) | Around the handler, before and after | `NestInterceptor` |
| Pipe | Validate and transform inputs | Just before the route handler runs | `PipeTransform` |
| Exception filter | Catch thrown errors and format the response | When an exception is thrown | `ExceptionFilter` |
